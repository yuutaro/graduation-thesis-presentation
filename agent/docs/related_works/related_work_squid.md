# 関連研究調査メモ: SQUiD (arXiv '25)

## 論文情報
- **タイトル:** SQUiD: Synthesizing Relational Databases from Unstructured Text
- **著者:** Mushtari Sadia, et al. (University of Michigan)
- **出典:** arXiv:2505.19025 (2025)
- **リンク:** [arXiv:2505.19025](https://arxiv.org/abs/2505.19025)

## 研究の概要
非構造化テキスト（自然言語のドキュメント）から、リレーショナルデータベース（スキーマ定義とデータ挿入）をゼロから自動構築するタスク「**Text2R**」を定義し、それを実現するフレームワーク「**SQUiD**」を提案した研究。
LLMに単に「DBを作れ」と指示するだけでは、外部キー制約の不整合やデータの欠損が発生するため、プロセスを以下の4段階に分解する手法をとる。

1.  **Schema Generation:** 文章からエンティティを特定し、正規化されたテーブル構造（PK/FK含む）を設計する。
2.  **Value Identification:** テキストから具体的な値（名前、日付、金額など）を抽出する。
3.  **Table Population:** 抽出した値をスキーマに合わせてタプル化し、テーブル間の参照整合性を解決する。
4.  **Database Materialization:** プログラム的に正しいSQL文を生成して実行する。

## 本研究（卒論）への導入・位置づけ

### 1. 共通する課題意識（Critical Shared Insight）
- **「LLMによる直接的なSQL生成の限界」**: 複雑なリレーションを持つDBを一度のプロンプトで生成しようとすると、整合性（特に外部キー制約）が破綻するという課題認識が完全に一致している。
- **「プロセスの分解」**: スキーマ設計とデータ生成を分ける、あるいはステップバイステップで生成するという解決策の方向性も共通している。

### 2. 決定的な違い（差別化ポイント）
「**抽出 (Extraction)**」か「**創造 (Generation)**」か、というベクトルの違い。

| 特徴 | SQUiD (Sadia et al., 2025) | 本研究 (卒論) |
| :--- | :--- | :--- |
| **入力** | 既存のテキスト（事実・記録） | 架空の要件・シナリオ（指令） |
| **目的** | 情報の構造化・正規化 | テストデータの創造・量産 |
| **データの元** | テキスト内に存在する事実 | LLMの知識と創造性による補完 |
| **評価軸** | 原文の情報をどれだけ正確にDB化できたか | 生成されたデータがどれだけ人間らしくリアルか |

### 3. 論文内での記述案（第2章 関連研究）

> 「テキストからのデータベース構築に関する研究として、Sadiaら[Ref]のSQUiDが挙げられる。彼らは非構造化テキストからリレーショナルデータベースを生成する際、LLMに直接SQLを出力させる手法では外部キー制約などの整合性維持が困難であることを指摘し、スキーマ設計とデータ投入のプロセスを分離するフレームワークを提案した。
> 本研究は、彼らの『プロセス分解による整合性担保』というアプローチを支持しつつ、それを『既存情報の抽出』ではなく『新規データの創造』に応用するものである。具体的には、Scenarist（脚本）による文脈生成と、Designer（設計）によるデータマッピングを分離することで、架空のデータであってもリレーションの整合性を厳密に保つことを可能にしている。」

## 補足
- この論文は「LLMでRDBを扱うなら、一発生成は無理だから分割しろ」というアーキテクチャの妥当性を裏付ける強力な援軍となる。
